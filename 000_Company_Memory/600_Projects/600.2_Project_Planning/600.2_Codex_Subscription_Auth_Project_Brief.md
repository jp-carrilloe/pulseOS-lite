## Goal
Allow PulseOS CLI chat and bootstrap flows to use a locally signed-in Codex session instead of requiring a manually pasted `OPENAI_API_KEY`.

## Why It Matters
The current CLI assumes provider API keys for all LLM access. That blocks users who are already paying for ChatGPT/Codex and are signed in locally. A Codex-backed OpenAI path reduces setup friction while preserving the existing API-key path for direct provider access.

## Scope
### In Scope
- Add a shared auth/provider abstraction for CLI model access.
- Support OpenAI auth strategies: `api_key`, `codex_cli_session`, `auto`.
- Use Codex session auth for OpenAI chat and bootstrap flows.
- Keep retrieval embeddings on API key or heuristic fallback in Phase 1.
- Update CLI status/help and docs to explain active auth mode and fallback behavior.

### Out of Scope
- Anthropic or Gemini non-API-key auth.
- Direct use of undocumented Codex credential files as the primary auth mechanism.
- Embedding access through Codex session auth.
- Cloud sync or remote task orchestration changes.

## Requirements
- Detect local Codex availability and login state using supported CLI commands.
- Preserve existing API-key flows.
- Fail with explicit messages when Codex is installed but not logged in, or not installed at all.
- Keep test coverage for auth resolution and fallback behavior.

## Key Decisions
- Phase 1 uses `codex exec` as the supported local transport for OpenAI chat/bootstrap.
- Phase 1 does not attempt Codex-backed embeddings.
- Local delivery tracking stays in `600_Projects`, not GitHub issues.

## Risks and Open Questions
- `codex exec` prompt formatting must stay deterministic enough for bootstrap output quality.
- Codex CLI command behavior may change across versions, so capability detection must stay narrow and defensive.
- If `codex exec` latency is materially worse than direct API calls, we may need prompt-shape tuning or provider-specific UX notes.
