## Purpose
Defines the generic CRM sync landing layer for company revenue data. This section is provider-neutral so HubSpot, Attio, Salesforce, Pipedrive, CSV exports, or future CRM connectors can write into one stable local shape before downstream agents use the data.

This document describes the operational contract only. API credentials, refresh tokens, and provider secrets must remain in `.env` or the relevant secret manager and must never be stored in Markdown or SQLite rows.

## Supported Object Types
The generic CRM mirror should accept these object types:

| Object Type | Purpose | Typical Downstream Use |
|---|---|---|
| `contact` | People, buyers, champions, influencers | Sales sequences, relationship history, attribution |
| `company` | Accounts, target organizations, customers | ICP analysis, segmentation, account planning |
| `deal` | Opportunities and revenue pipeline | Forecasting, pipeline velocity, conversion reporting |
| `activity` | Calls, emails, meetings, notes, tasks | Engagement history and sales-process quality |
| `ticket` | Support or delivery issues | Delivery handoff and churn-risk analysis |
| `custom` | Provider-specific objects that do not map cleanly | Future extension without schema churn |

## SQL Landing Tables
The local CLI database creates two CRM tables:

| Table | Role |
|---|---|
| `crm_objects` | Provider-neutral mirror of CRM records with normalized query fields plus `raw_json`. |
| `crm_sync_runs` | Audit trail for sync attempts, source provider, status, row counts, and errors. |

### `crm_objects`
Each CRM row is uniquely identified by:

```text
provider + object_type + provider_object_id
```

Normalized fields include provider, object type, provider object id, name, email, company name, lifecycle stage, pipeline stage, owner, amount, currency, close date, last synced timestamp, and raw provider JSON.

### `crm_sync_runs`
Each sync run records provider, sync type, status, start and completion time, objects seen, objects upserted, and any error text. This table stores operational metadata only and must not store credentials.

## Sync Expectations
- Sync implementations should upsert into `crm_objects` rather than append duplicates.
- Provider-specific fields that do not have a normalized column should remain in `raw_json`.
- Failed syncs should write a `crm_sync_runs` row with `status = failed` and a concise error.
- Partial syncs should preserve existing CRM rows unless the connector has an explicit deletion/archive policy.
- Sync timestamps should use ISO 8601.

## Privacy And Data Handling
CRM data can include personal data, customer information, commercial notes, and sensitive account context. Treat `crm_objects` as sensitive operational data.

Rules:
- Do not sync API keys, OAuth tokens, passwords, private notes unrelated to sales operations, or payment details.
- Keep access to the SQLite database local and controlled.
- Use `raw_json` for source fidelity, but avoid storing fields that violate the source CRM's privacy or retention policy.
- Confirm legal/privacy requirements with @Legal before adding production CRM syncs for regulated industries or EU personal data.
