## 3. Vendor Evaluation Criteria

Before adopting any new tool or vendor:

| Criteria | Questions |
|---|---|
| Fit | Does it solve a real, recurring problem? Is there overlap with an existing tool? |
| Cost | What is the true all-in cost (setup, per-seat, integration effort)? |
| Security | Where is data stored? Does it meet our data handling requirements? (Ref: `105_Technical_Infrastructure_and_Security`) |
| Integration | Does it integrate cleanly with our stack, or does it create a silo? |
| Vendor health | Is the company well-funded and stable? What does the exit look like? |
| Contract terms | Auto-renew clauses? Data portability on exit? (Ref: `106_Legal_and_Compliance`) |

**Adoption Process:**
1. Proposal written by requesting team member
2. Operations Agent reviews against criteria
3. Finance approval for tools above $[X]/month
4. Security review for tools handling client or sensitive data
5. Added to registry on Day 1 — not after

## 4. Vendor Offboarding Process

When decommissioning a tool or vendor:

- [ ] Export all data before access is revoked
- [ ] Confirm data deletion from vendor's systems (get written confirmation)
- [ ] Revoke all user access and API keys
- [ ] Cancel billing — confirm cancellation receipt
- [ ] Update registry status to "Offboarded"
- [ ] Document reason for switch (institutional knowledge)
